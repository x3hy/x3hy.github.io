  • TRCS in-depth breakdown 2026-06-11T23:47:38.144Z agoTRCS is a mostly-single-file 400 line program written in ANSI C99. Connection loop: TRCS sets up the socket connection before starting a basic client-accept loop, this loop will basically just await a new client connection. If you're unaware, when I refer to a "client" I refer to any incoming connection to my server, this could refer to a person connecting to the server on their browser or a person pinging the server locally, just to name some examples. What are headers? Headers are used as per-request configurations, you can use `curl -i ...` to print the HTTP headers when connecting to an address. The server returns certain headers to indicate return statuses too. For example, if you connect to my server it might return "200 OK" which signifies a successful connection, other more known http codes are 404 (not found) and 500 (internal error). Thread stage1: When a client connects to the server a new thread is created which is given the unique client_fd of the connecting client. When connecting to the server, clients send a bunch of information including the http endpoint requested. Here is an example of what the server receives in HTTP headers if you where to connect to "localhost:6060/hello/world": GET /hello/world HTTP/1.1 Host: localhost:6060 User-Agent: curl/8.20.0 Accept: */* Back to the client thread, the threads one job is to filter through the clients headers and fetch the endpoint, in this case the endpoint is "/hello/world". this endpoint is then passed off to the parsing function "thread_handle_path()", this function splits the endpoint into seperate parts. Lets assume that we are iterating through the parts of the endpoint, "hello" being the first part and "world" being the second part. My server expects the following endpoint format, the first part should be either "sock" to indicate the client wants to start a stream, or "post" to indicate the client is trying to post a message. The second part must be the ID of the server, when stating the server you are required to provide an ID as the last argument, this ID most be given in the second part of the request, this is used asa simple form of authentication, it is not very secure but it works for now, later on I'll write a rolling key system that would work in its place. Now for the third part, if the client is trying to post a new message an additional third part must be given which should contain the URL-safe text content that will be added as a new message on the server. When the thread_handle_path() function is called, it will first claim its thread, this refers to the system in place that limits the amount of total clients/threads that can connect to the server at one time. This works by storing an array of available threads with each value being a dummy value of -1, (this value is referenced as THREAD_EMPTY in the code). When a thread wants to claim a place in the thread array, it will search for the first index in the array that is equal to THREAD_EMPTY, this index is then set to a value that is not THREAD_EMPTY to show that that item in the array is allocated to a thread. The messaging system / posts: To send a message the client must connect to /post with the appropriate part format, when this is authenticated (through thread_handle_path()) the given message will be appended to a char* array called "backlog" this backlog holds a set amount of messages, by default this is ten. The backlogs job is to work as a sort of temporary queue to hold messages if multiple messages where sent at the same time. If instead I just relayed incoming messages directly to the client threads, because the threads only check for new messages every so often (and not all the time which would be impossible for computational and resource limits) it means that if messages where sent within the same interval there is a large chance that one of these messages will be overwritten in memory before the client threads are able to relay it back to the client. This backlog solves this issue by holding the last 10 messages which means that if threads check for new messages every one second, a maximum of 10 messages can be sent and handled correctly at one time. You might be wondering "What about if there is more then 10 messages sent at one time?" then messages will overflow and yes messages will be lost, to combat this indefinitely I added a MAX_THREAD value which limits the amount of clients that can connect to the server at one time. If this value is equal to ten it means that if every user that could possibly be connected sent a message at the exact same time, then the server would be able to handle this fine. Streaming: TRCS uses a basic SSE (Server-Send-Events) system for maintaining a constant connection from a client to the server. This is very easily done using the pre-defined HTTP headers from before, you just need to include the following options for it to start as a SSE: Content-type: text/event-stream Cache-Control: no-cache, no-store, must-revalidate Connection: keep-alive Back to the thread claiming system, the reason the thread needs to claim a thread is not just for thread limiting but also for message syncing, the thread will update its index integer with the total amount of messages that thread has sent to the client. In the TRCS config file you can define how many times a thread checks for new messages a second, the default is once every second. When a thread checks for new messages it simply removes the threads total message count by a server-stored, if the resulting value is not equal to 0 it means that the thread is behind, when this occurs the thread will then loop through the backlog and print out the messages that it missed. When a client exits the thread detects this (when the thread sends data to the streaming client and it fails it means the client has closed the connection) the thread sets its index in the threads array back to THREAD_EMPTY so that another thread can open up and use it. One more thing to note is that if the user is POSTING a message, the thread will not check to see if there is an empty thread, this will only occur on stream connections. Again, like the last post, if you have any questions, email me using the contact information above. This project is still a work in progress so if you experience any issues please create an issue on github or again, email me. Thank you for reading!
